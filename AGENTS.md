# AGENTS.md

## 1. 总纲：项目不是“普通毕业设计”，而是“作品级数字体验项目”

本仓库用于实现一个可运行、可演示、可持续迭代的数字文旅体验项目，但它的目标标准不是“学生作业做完整”，而是：

- 作为作品成立
- 作为产品原型成立
- 作为设计系统成立
- 作为前后端工程实现成立
- 作为 AI 融合体验成立

默认前提：

1. 不要把项目当成“普通学生项目”来写。
2. 不要用“能跑就行”“够答辩就行”“先凑出来”这种低标准思维输出结果。
3. 任何设计、前端、后端、AI 改动，都必须以“作品级完成度”为目标。
4. 默认项目负责人具备高审美、高标准、强产品判断、强工程要求；不得擅自降低身份预设。
5. Codex 必须以“世界一流设计家 / 艺术家 / 顶尖工程师 / 顶级 PM 协同体”的标准思考与实施，而不是以“编程基础一般的学生”作为默认用户画像。

最高优先级：

- 做出有记忆点、有导演意识、有叙事张力、有工程质量的完整作品
- 让前后端和 AI 一起为前台体验服务
- 在保证可运行闭环的前提下，把质量上限尽可能拉高

---

## 2. 核心价值排序

本仓库后续所有决策，默认遵守以下优先级：

1. **体验质量与作品完成度**
2. **叙事、信息架构与视觉表达的一致性**
3. **工程质量、可维护性、性能和稳定性**
4. **真实数据流动与前后端闭环**
5. **实现成本与开发便利性**

解释：

- “容易写”不能凌驾于“结果高级”之上。
- “能跑”只是底线，不是目标。
- “便于答辩”不能成为牺牲设计与工程质量的借口。
- 工程实现不是为了堆技术，而是为了支撑更强的前台体验。
- 如果一个方案只是更省事，但明显降低最终气质，应优先选择更高质量方案。

## 3. 项目定位

项目定位不是传统旅游信息站，也不是民宿预订平台，更不是普通模板化展示网站。

这是一个：

- 中文旅游文化叙事站
- 数字长卷式内容体验站
- 具有策展意识与导览感的文旅作品
- AI 深度融入的文化体验产品原型

页面目标不是“把信息展示出来”，而是让用户：

- 被气质吸引
- 被叙事带入
- 被章节引导
- 被图像记住
- 被 AI 陪同继续探索

项目必须避免以下错误定位：

- 普通学生项目
- 普通政务旅游门户
- 普通资讯列表站
- 普通白底卡片站
- 只有聊天框的 AI 装饰功能页

## 4. 设计与体验总原则

### 4.1 设计原则

1. 页面必须有导演意识，而不是模块堆砌。
2. 首屏必须承担情绪和世界观建立功能，而不是只承担导航入口功能。
3. 图像是结构骨架，不是装饰背景。
4. 页面必须有章节感、呼吸感、节奏感，而不是一屏接一屏均匀堆叠。
5. 每个页面必须有 dominant idea，不允许所有区块同权。
6. 不接受廉价“高级感”：不接受靠渐变、毛玻璃、重阴影、发光、模板动画制造伪高级。
7. 高级感必须来自：
   - 真图与裁切控制
   - 排版层级
   - 留白与构图
   - 色彩节制
   - 题注与章节系统
   - 精准动效
   - 工程打磨痕迹

### 4.2 页面角色原则

- `HomeView`：片头海报 / 宣言页 / 世界观入口
- `TopicListPage`：章节序厅
- `TopicDetailPage`：馆藏解说页
- `ScenicListView`：地方图谱页
- `ScenicDetailView`：地方 dossier
- `AiChatView`：随行讲解员
- `AiTripView`：路线工作室 / 路线长卷
- `AboutView`：策展声明与方法页
- `SiteLayout`：展览外壳，不是普通门户壳

不允许把所有页面都做成“标题 + 简介 + 卡片 + CTA”的同构模板。

### 4.3 视觉系统原则

1. 优先建立完整、语义化、可扩展的 design token 系统。
2. 样式系统必须先于页面大规模扩写。
3. 样式命名必须统一、有层次、有职责边界。
4. 允许页面有局部特化，但不允许页面各自定义一套视觉语言。
5. 组件系统必须服务叙事，而不是让所有内容都被组件模板抹平。
6. 无障碍、响应式、焦点态、触控目标、文本缩放是基础要求，不是附加项。

### 4.4 动效原则

1. 动效是工艺表达，不是装饰。
2. 动效必须传达节奏、气口、镜头感、颗粒度和页面被认真打磨过的感觉。
3. 页面动效要像导演调度，不像组件库统一播放。
4. 高级动效应当：
   - 克制
   - 精准
   - 低噪声
   - 分层
   - 有节奏
   - 有呼吸感
5. 禁止廉价动效：
   - 全站统一 fade-up
   - 所有卡片 hover 上浮
   - 夸张弹跳
   - 大面积 blur
   - 无意义旋转
   - 科技蓝紫霓虹
   - 滚动时元素乱飞
   - 模板式 reveal 套餐
6. 优先动画 `opacity` 与 `transform`，严格控制性能成本。

## 5. 工程与代码标准

### 5.1 默认质量门槛

1. 默认输出应达到高水准，不允许因为“项目是毕业设计”就降低代码和设计质量。
2. 代码必须清晰，但“清晰”不等于“幼稚”。
3. 不允许以“便于初学者理解”为理由牺牲架构质量、视觉表达和实现精度。
4. 每一行代码都必须对结果负责：
   - 有职责
   - 有边界
   - 有一致性
   - 有性能意识
5. 不接受“能跑但粗糙”的最终实现。

### 5.2 工程原则

1. 优先单一职责和清晰分层。
2. 优先可验证、可维护、可扩展。
3. 优先真实数据闭环。
4. 允许抽象，但必须有价值；不允许无意义炫技式抽象。
5. 不允许让技术实现绑架体验目标。
6. 不允许为了偷懒而让页面退回低级模板化实现。
7. 性能优化必须贯穿实现过程，而不是最后补救。

### 5.3 前端标准

1. 前端不是“把接口数据显示出来”，而是把内容编排成体验。
2. 必须优先考虑：
   - 视觉层级
   - DOM 语义
   - 响应式结构
   - 动效时序
   - 图像裁切
   - 内容可读性
   - 移动端表现
3. 页面实现要有“作品感”，不是只追求“页面整齐”。
4. Element Plus 只保留功能价值，不允许让默认视觉主导公共页面。
5. 公共页面必须有自己的设计语言、自己的动效语言、自己的组件语言。

### 5.4 后端标准

1. 后端不仅提供 CRUD，还必须为前台体验提供叙事支持。
2. 优先在服务层组织前台视图模型，必要时再推进数据库正式落库。
3. 接口要为前台编排服务，而不是把数据库字段原样丢给前端。
4. 允许为了更好的前端体验扩充响应结构。
5. 后端改动必须兼顾：
   - 路由稳定性
   - 数据一致性
   - 模型同步
   - SQL 真源
   - 演示闭环

## 8. 不允许做的事情

1. 不允许擅自降低项目标准。
2. 不允许默认按“普通学生项目”写实现。
3. 不允许只做静态页面冒充体验重构完成。

13. 不允许引入破坏可维护性的黑魔法代码。
14. 不允许用“高级感”作为借口堆砌无意义特效。
15. 不允许把 AI 页面继续做成外挂工具页。

## 9. 命名规范与接口纪律

### 前端命名

1. Vue 页面组件使用 PascalCase，例如 `ScenicListView.vue`
2. 通用组件使用 PascalCase
3. store 使用 `useXxxStore`
4. 路由 path 使用 kebab-case
5. API 文件按模块命名
6. 样式类名必须统一、有层级、有语义，不允许杂乱缩写
7. 建议区分：
   - layout：`l-*`
   - component：`c-*`
   - pattern：`p-*`
   - utility：`u-*`
   - state：`is-*` / `has-*`

### 后端命名

1. 路由、控制器、服务按模块拆分
2. 数据表名使用复数下划线风格
3. 数据库字段名使用 snake_case
4. JS 变量与函数名使用 camelCase
5. 常量使用大写下划线命名

### 接口返回规范

所有接口统一返回：

成功：
```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
失败：
```json
{
  "code": 500,
  "message": "错误信息",
  "data": null
}

约束：

1.前台接口统一前缀为 /api/front
2.后台接口统一前缀为 /api/admin
3.分页列表接口统一返回 list、total、page、pageSize
4.不允许返回结构风格混乱
5.token 校验失败必须统一处理
6.对前台体验重要的接口，应优先提供结构化、可编排的数据模型

## 10. 数据模型与数据库变更规范

1. `sql/schema.sql` 是数据库结构真源。
2. 任何表结构变更，必须同步更新：
   - SQL 脚本
   - Sequelize 模型
   - 接口文档
   - 后台表单
   - 前台展示消费逻辑
3. 优先采用“新增字段优先，少做破坏性变更”。
4. 每次改表必须检查 seed 是否仍可用。
5. 所有核心表必须有基础索引。
6. 删除字段前必须检查前台、后台、AI prompt、服务层视图模型依赖。
7. 对前台体验重要的字段，优先考虑：
   - 叙事字段
   - 媒体字段
   - 裁切字段
   - 焦点字段
   - 关联关系字段
   - 路线/章节字段

建议重点支持的前台叙事字段包括但不限于：

## 11. AI 模块实现标准

AI 模块必须是站点体验的一部分，而不是装饰聊天框。

### AI 问答

1. 必须先从 `scenic_spots` 与 `articles` 中召回上下文。
2. 首版可用 `like` 或等价模糊检索，不引入向量数据库。
3. 必须将“用户问题 + 召回上下文 + 角色约束”一起构造成 prompt。
4. 必须记录 `ai_chat_logs`。
5. 返回结构优先支持前台编排，不应只返回一段原始 answer。
6. 推荐支持：
   - `leadTitle`
   - `answerBlocks[]`
   - `citations[]`
   - `relatedCards[]`
   - `followupPrompts[]`
   - `heroSpotlight`

### AI 行程

1. 必须结合 `days`、`preferences`、`departureArea`、`pace`、`extraRequirement` 组织 prompt。
2. 必须先从数据库筛选候选内容，再生成结果。
3. 结果必须可演示、可编排、可理解。
4. 必须记录 `ai_trip_logs`。
5. 推荐支持：
   - `routeTitle`
   - `routeMood`
   - `introNote`
   - `days[].chapterTitle`
   - `days[].coverSpot`
   - `days[].items[].visualHint`
   - `packingTips`
   - `routeWarnings`
   - `citations[]`

### AI 文案生成

1. 只服务于内容运营场景。
2. 输出必须允许管理员编辑后再保存。
3. 必须记录 `ai_copywriting_logs`。

### Prompt 管理

1. 所有 prompt 模板必须放在 `server/src/prompts/`
2. 不允许把长 prompt 直接硬编码在控制器里
3. prompt 必须体现赣州旅游文化语境
4. prompt 必须服务结构化输出，而不是只追求模型自由发挥

### 稳定性

1. AI 接口失败必须有友好兜底
2. 模型超时、无 key、无上下文等都必须可控处理
3. AI 不能成为平台单点故障
4. AI 结果必须尽量可验证、可引用、可继续探索

## 12. 素材、媒体与占位标准

1. 真实图像优先于装饰图形。
2. 没有强图时，宁可使用设计化章节占位板，也不要使用廉价渐变占位图。
3. 所有重要媒体都应考虑：
   - 桌面裁切
   - 移动裁切
   - 焦点位置
   - 题注位置
   - alt text
   - 来源与 credit
4. 页面至少应区分：
   - 建立场域的大图
   - 解释气质的中图
   - 强化记忆的细节图
5. 视频只作为增强项，不可依赖低质量素材硬上。

## 13. 性能与稳定性标准

1. 默认把性能当成作品质量的一部分。
2. 所有前端实现必须考虑：
   - 首屏渲染
   - 图像尺寸
   - 动效成本
   - 滚动流畅度
   - 移动端性能
3. 默认优先动画 `opacity` / `transform`
4. 不允许大面积实时 blur、重滤镜、无意义高成本动画
5. 不允许让炫技效果破坏滚动与交互稳定性
6. 如果某个效果高级但不稳，优先舍弃
## 获准采用的高级前端技术栈

本项目允许引入经过筛选的现代前端体验技术，前提是这些技术能显著提升创作体验、沉浸感、叙事节奏或动态效果质量。

获准采用的一线选项：
- GSAP + ScrollTrigger：主要动态协调层
- View Transition API：在支持的情况下实现轻量级的路由/状态连续性
- Lenis：仅当滚动手感与同步性能实质性提升体验时，才可选用的平滑滚动层
- TresJS：仅用于本地沉浸式主角/氛围/场景层的增强

有条件允许：
- Rive：仅限用于孤立的高价值交互式动态组件
- 原生 Three.js：仅当 TresJS 的抽象层无法满足特定场景且理由充分时

不可妥协的规则：
- 不得仅为追求“更高端”的外观而引入库
- 不得构建包含多个相互竞争的动画系统的碎片化动态技术栈
- 不得将 WebGL 设为全站依赖项
- 除非有充分理由，否则不得在内容阅读区域使用高负载交互技术
- 每个新依赖项必须通过以下内容进行论证：
  1. 解决的具体体验问题
  2. 现有技术栈为何不足
  3. 性能开销
  4. 受影响的页面
  5. 降级策略

优先级：
- 保持可维护性、节奏感及原生体验
- 优先采用精准、局部的增强方案，而非追求宏大的技术炫技

1. 确切的用户体验提升

## 14. 自检与验收标准

每次完成一个功能点或一个页面，必须至少自检以下事项：

### 工程自检
1. 是否破坏统一接口格式
2. 数据库、模型、接口、前台展示是否一致
3. 是否有明显未导入、路径、命名错误
4. 是否仍然满足可运行闭环
5. 是否同步了文档与配置说明

### 设计自检
1. 页面是否仍然像模板站
2. 首屏是否承担世界观建立功能
3. 图像是否成为结构骨架
4. 页面节奏是否有层次
5. 页面之间是否属于同一世界
6. 动效是否高级、克制、讲究
7. 移动端是否仍然保有气质
8. 是否存在廉价占位、廉价渐变、廉价 hover

### AI 自检
1. 是否有数据库召回
2. 是否有日志记录
3. 是否返回结构化内容
4. 是否真正服务导览体验
5. 是否能在后台看到相关记录

### 演示自检
1. 首页 5 秒内能否让人感到“不是普通旅游站”
2. 主题页之间是否一眼可区分
3. 景点详情首屏是否先记住地方，而不是先看到字段
4. AI 页是否像体验的一部分，而不是外挂工具
5. 无图状态是否仍然体面

## 15. 默认输出方式要求

Codex 在本仓库中默认不得输出低级、保守、模板化方案。

默认输出要求：

1. 先判断当前阶段，再决定做法
2. 先给结构判断，再写实现
3. 先考虑页面角色、体验节奏、素材与数据支撑，再写 UI
4. 不要默认保留旧结构
5. 不要默认“更整齐”就是更好
6. 不要默认“更容易讲解”就是更优解
7. 需要大胆时必须大胆，需要克制时必须克制
8. 所有实现都应尽量接近作品级结果，而不是作业级结果
